# Celebration-анимация при достижении цели недели

## Контекст и цель

Спек [2026-07-18-week-goal-tile-design.md](2026-07-18-week-goal-tile-design.md) ввёл
плитку «цель недели» (`weekWorkouts / weekGoal`) со статичным состоянием
«достигнута» (акцентная заливка, `✓`, полностью залитый прогресс-бар) и явно
отметил в разделе «Дальнейшее развитие»: «При достижении `weekGoal` — haptic +
короткая анимация заливки плитки». Персональная цель (спек
[2026-07-19-weekly-goal-personalization-design.md](2026-07-19-weekly-goal-personalization-design.md))
уже реализована — это следующий шаг из того же списка.

**Область этого спека:** одноразовая анимация + haptic-отклик в момент, когда
пользователь впервые за неделю видит плитку в состоянии «цель достигнута». Не
входит: изменения бэкенда/схемы БД, экран настроек, что-либо кроме визуального
и тактильного отклика на уже существующей плитке.

## Архитектурная развилка (решено)

Тренировки создаются **только через бота** в чате, а не в Mini App — значит
момент «цель только что закрыта» в реальном сценарии почти всегда совпадает с
**холодной загрузкой** Mini App (пользователь залогировал тренировку в чате,
затем открыл приложение), а не с живым переходом `weekWorkouts` через порог
внутри одной сессии React-компонента. Чистое сравнение «было < цели, стало ≥
цели» через `useRef`/предыдущий рендер почти никогда не сработает на практике.

**Решение:** дедупликация через `localStorage`, ключ по понедельнику текущей
недели (`gym95:goal-celebrated:<YYYY-MM-DD>`). Анимация играет при первом
рендере (холодном или живом), где плитка оказывается в состоянии «достигнута»
и флag для этой недели ещё не стоит — затем флаг ставится, и до смены недели
анимация больше не повторяется, сколько раз ни открывай приложение.

Вся логика инкапсулируется в `SummaryBar.tsx` — этот компонент уже единолично
владеет рендером плитки и вычисляет `reached`. `App.tsx` и бэкенд не
затрагиваются.

## 1. Триггер и дедупликация

- В `SummaryBar.tsx` — приватная функция `weekMondayKey()`: понедельник текущей
  календарной недели по **локальной** дате клиента (не обязана побитово
  совпадать с UTC-версией `mondayKey` на бэкенде — используется только как ключ
  дедупликации в браузере, не для сверки данных).
- `localStorage` ключ: `gym95:goal-celebrated:${weekMondayKey()}`.
- Новое состояние `const [celebrating, setCelebrating] = useState(false)`.
- `useEffect` с зависимостями `[summary.weekWorkouts, summary.weekGoal]`:
  вычисляет `reached` (переиспользует существующую переменную/логику), и если
  `reached && localStorage.getItem(key) !== '1'` — вызывает
  `setCelebrating(true)`, `notifyHaptic('success')` (уже используется в проекте
  для success-момента, см. `WorkoutEditScreen.tsx`), затем
  `localStorage.setItem(key, '1')`.
- `celebrating` сбрасывается обработчиком `onAnimationEnd` на самой плитке —
  React-класс `celebrate` снимается сразу после того, как CSS-анимация
  доиграла один раз, никакого дополнительного таймера не требуется.
- `localStorage.getItem`/`setItem` оборачиваются в `try/catch` — если хранилище
  недоступно (приватный режим, редкий edge-case вебвью), деградация
  допустима (анимация может играть чаще одного раза за неделю), но рендер
  плитки не должен падать.

## 2. Анимация (блик-развёртка)

- Новый CSS-класс `.stat-tile.celebrate`, применяется **вместе** с уже
  существующим `.reached` (тот не меняется — после того как анимация
  доиграла, плитка остаётся в обычном статичном виде «достигнуто»).
- `.stat-tile.celebrate` — `animation: celebrateGlow 1.4s ease-out;`
  (усиленное свечение границы через уже существующую переменную `--glow`,
  играет один раз, не `infinite`).
- `.stat-tile.celebrate::after` — диагональный блик
  (`linear-gradient(115deg, transparent, rgba(255,255,255,0.35), transparent)`),
  `animation: celebrateSweep 1.4s ease-out;`, синхронная длительность с
  `celebrateGlow`.
- Оба keyframes — одноразовые версии анимаций, показанных на мокапе (вариант
  B): без «холостого хода» 0–60%, которым в мокапе имитировалось цикличное
  повторение для наглядности — здесь анимация идёт от 0% до 100% сразу.
- `--glow` в светлой теме — `none` (существующее поведение, не меняется):
  свечение видно только в тёмной теме Telegram, согласно уже принятому в
  проекте принципу «яркие акценты — только в тёмной теме». В светлой теме
  обратная связь всё равно есть через haptic и смену цвета границы плитки
  (не зависит от `--glow`).

## 3. Обработка ошибок и краевые случаи

- Открытие Mini App через несколько дней после того, как цель уже была
  отпразднована на этой неделе → флаг в `localStorage` уже стоит → анимация не
  играет, плитка отображается в обычном статичном состоянии «достигнуто».
- Наступление новой календарной недели → `weekMondayKey()` меняется → флага
  для нового ключа ещё нет → при следующем достижении цели анимация сыграет
  заново.
- Пользователь удаляет тренировку, из-за чего `weekWorkouts` опускается ниже
  цели, а затем снова её достигает в течение той же недели → анимация повторно
  **не** играет (флаг стоит на всю неделю целиком, не на конкретное
  пересечение порога) — сознательное упрощение, задокументировано здесь как
  принятое поведение, не баг.
- `weekGoal <= 0` → `reached` остаётся `false` благодаря уже существующей
  защите в `SummaryBar.tsx` — celebration не триггерится, поведение не
  меняется.
- `localStorage` недоступен → см. п. 1 (try/catch, деградация без краша).

## Файлы к изменению

- `webapp/src/components/SummaryBar.tsx` — `weekMondayKey()`, состояние
  `celebrating`, `useEffect`-триггер, `onAnimationEnd`, применение класса
  `celebrate`.
- `webapp/src/styles.css` — `.stat-tile.celebrate`, `::after`, keyframes
  `celebrateGlow`/`celebrateSweep`.

Изменений в бэкенде, схеме БД или контрактах API не требуется.

## Тестирование

Формальных тестов в проекте нет — ручной прогон (dogfood), как и для всех
предыдущих фич:

1. Очистить `localStorage` (или открыть в приватном/инкогнито-окне) → состояние
   с `weekWorkouts >= weekGoal` → анимация играет один раз при открытии.
2. Повторно открыть/обновить страницу в течение той же недели → анимация не
   повторяется, плитка в обычном статичном состоянии «достигнуто».
3. Проверить в светлой и тёмной теме Telegram — граница/haptic работают в
   обеих, свечение — заметно только в тёмной (ожидаемо, не баг).
4. Собрать webapp (`npm run build` в `webapp/`) — сборка зелёная.

## Вне области этого спека

- Изменения бэкенда/БД — дедупликация полностью клиентская.
- Экран настроек для управления celebration (вкл/выкл) — не запрашивалось.
- Повторное празднование при повышении объёма/перевыполнении цели в течение
  той же недели (`4 / 3` после уже отпразднованных `3 / 3`) — не в области,
  текущее поведение (без анимации) сохраняется автоматически за счёт
  дедупликации по неделе.
