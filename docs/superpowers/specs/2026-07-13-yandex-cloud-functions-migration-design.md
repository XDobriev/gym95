# Перенос хостинга бота с Railway на Yandex Cloud Functions

## Контекст и цель

gym95 — личный Telegram-бот (Node.js/TypeScript, Telegraf), сейчас развёрнут на Railway в режиме long polling (`bot.launch()`). Требования пользователя: хостинг должен быть бесплатным и не требовать иностранной карты — Railway этому не соответствует. Yandex Cloud Functions удовлетворяет обоим условиям: постоянный бесплатный лимит на вызовы и вычислительные ресурсы, аккаунт активируется российской картой.

**Область этого спека:** перенос только процесса бота. База данных (Supabase) не трогается — остаётся как есть. Возможный последующий переезд БД на YDB — отдельный спек, вне области этой работы.

## Архитектура

Прямой HTTP-триггер функции, без API Gateway.

У Yandex Cloud Functions есть встроенный публичный HTTPS-эндпоинт (`functions.yandexcloud.net/<id>`) с опцией разрешить вызов без IAM-авторизации ("публичная функция"). Этого достаточно, чтобы Telegram мог напрямую стучаться в вебхук. API Gateway для одного вебхука — лишняя сущность, не даёт ничего взамен добавленной сложности.

Рассмотренная и отклонённая альтернатива: держать polling внутри функции, запускаемой по таймеру раз в минуту. Плохо согласуется с моделью Telegram (`getUpdates` и `setWebhook` взаимоисключающие) и добавляет задержку до минуты на сообщение.

**Поток данных:** Telegram → POST на публичный URL функции → Yandex Cloud Functions поднимает Node.js 22 контейнер → `handler(event, context)` парсит `event.body` → `bot.handleUpdate(update)` → ответ `{statusCode: 200}`.

Node.js 22 выбран, т.к. это единственная поддерживаемая и рекомендуемая версия runtime в Cloud Functions на данный момент, и она совпадает с текущим требованием `engines.node >= 22` в `package.json` — переписывать под другую версию не нужно.

## Изменения в коде

Сейчас создание `Telegraf`-инстанса и регистрация хендлеров (`registerStart`, `registerWorkout`, `registerHistory`, `registerProgress`, `registerExport`) зашиты в `src/bot.ts` вместе с вызовом `bot.launch()` и обработкой `SIGINT`/`SIGTERM`.

Изменения:

1. **`src/botInstance.ts`** (новый) — создание `Telegraf` инстанса, регистрация всех хендлеров, `bot.catch(...)`. Ничего не запускает — просто экспортирует готовый `bot`.
2. **`src/bot.ts`** (изменяется) — импортирует `bot` из `botInstance.ts`, оставляет только `bot.launch()` + graceful shutdown. Используется для локальной разработки (`npm run dev`, polling).
3. **`src/webhook.ts`** (новый) — точка входа для Yandex Cloud Functions:
   ```ts
   import { bot } from './botInstance';

   export const handler = async (event: { body: string }, context: unknown) => {
     try {
       const update = JSON.parse(event.body);
       await bot.handleUpdate(update);
     } catch (err) {
       console.error('Ошибка обработки вебхук-апдейта:', err);
     }
     return { statusCode: 200, body: '' };
   };
   ```
   Отвечает `200` даже при внутренней ошибке — иначе Telegram будет бесконечно ретраить апдейт.

`config.ts` не меняется — секреты уже читаются через `process.env`, `dotenv.config()` в облаке просто не найдёт `.env`-файла и не помешает.

## Конфигурация и секреты

`BOT_TOKEN`, `SUPABASE_URL`, `SUPABASE_KEY` задаются как переменные окружения функции в консоли/CLI Yandex Cloud. Хранение секретов в Yandex Lockbox рассматривается как возможное будущее усиление, не обязательное для MVP.

## Сборка и деплой

- Бандлинг `src/webhook.ts` со всеми зависимостями в один файл через **esbuild** — уменьшает размер архива и cold start по сравнению с заливкой всего `node_modules`.
- Деплой через `yc` CLI: `yc serverless function version create ... --source-path <zip>`.
- Функция создаётся с публичным (неавторизованным) доступом на вызов.
- `railway.json` и `Procfile` удаляются как часть этой миграции — хостинг полностью переезжает с Railway.

## Первоначальная настройка вебхука

После первого деплоя функция получает постоянный URL. Разовым шагом (ручная команда или маленький npm-скрипт) вызывается `bot.telegram.setWebhook(<url функции>)`, чтобы Telegram начал слать апдейты на новый адрес.

## Тестирование

Формального набора тестов в проекте нет. План проверки после деплоя:
1. Задеплоить функцию, получить публичный URL.
2. Вручную вызвать URL с тестовым JSON апдейтом (curl/Postman) — убедиться, что возвращается 200 и в логах функции видна обработка.
3. Выполнить `setWebhook`.
4. Отправить боту реальное сообщение в Telegram, убедиться, что все существующие команды (`/new_workout`, `/history`, `/progress`, `/export`) работают как раньше.
5. Проверить `getWebhookInfo` — убедиться, что нет накопленных ошибок доставки.
